01

Layered Architecture

UI、Application、Domain、Infrastructure のように責務を層に分け、上位が下位を使う形にする。

UI / Presentation
Application
Domain
Infrastructure
skip layer = coupling

動き: 依存は上から下へ流れる。下層を飛ばした呼び出しが増えると層の意味が薄れ、結合が強くなる。

利点

  • 責務分離が分かりやすく、一般的な Web 業務アプリに導入しやすい。
  • 変更箇所を層単位で探しやすく、学習コストが比較的低い。

トレードオフ

理解容易性高
過剰抽象中
境界侵食中

相性の良いシステムアーキテクチャ

3-Tier / N-TierMonolithic SystemClient-Server

コード上の層と配置上の層を対応させやすく、単一アプリでも責務分離の説明がしやすい。

02

Clean Architecture

ドメインやユースケースを中心に置き、外側の UI、DB、Framework から中心へ依存させる。

Domain
Frameworks
Adapters
Use Cases
dependency inward

動き: 入出力は外側で変換され、依存は中心へ向かう。DB や UI が変わっても中心の業務規則を守る。

利点

  • 業務規則をフレームワークや外部サービスから独立させやすい。
  • 中心部分をユニットテストしやすく、長寿命のドメインに向く。

トレードオフ

独立性高
初期手数高
形式先行リスク中

相性の良いシステムアーキテクチャ

MicroservicesHybrid / DistributedServerless

外部通信、DB、クラウドサービスを外側に隔離できるため、配置や実行基盤が変わっても中心を保ちやすい。

03

Hexagonal Architecture

アプリケーションの中心にポートを置き、外部の DB、API、UI、メッセージングを adapter として接続する。

Application Core
Inbound Port
Outbound Port
Web UI
CLI
DB
API
core sees ports

動き: 複数の入口と出口がポートを通じて中心へ接続する。adapter を差し替えても中心の契約は維持する。

利点

  • 外部技術の差し替えとテストダブルの利用がしやすい。
  • 外部サービスの都合をドメインロジックへ漏らしにくい。

トレードオフ

差し替え容易高
adapter 数高
契約設計負荷高

相性の良いシステムアーキテクチャ

MicroservicesSOAHybrid / Distributed

外部システムや通信方式が多い構成で、接続方式を adapter として隔離できるため変更に耐えやすい。

04

MVC

Model、View、Controller に分け、ユーザー入力、状態、表示の責務を整理する UI 中心の構造。

Controller
Model
View
input
render

動き: 入力は Controller が受け、Model の状態を変え、View が表示を更新する。表示責務と状態責務を分ける。

利点

  • Web アプリや GUI で、入力処理、状態、表示の責務を説明しやすい。
  • フレームワークの標準構造として採用されることが多い。

トレードオフ

UI 整理高
Controller 肥大高
ドメイン保護中

相性の良いシステムアーキテクチャ

Client-Server3-Tier / N-TierMonolithic System

同期リクエストと画面生成の流れが明確な構成で、MVC の制御フローと合わせやすい。

05

MVVM

View と ViewModel をデータバインディングでつなぎ、表示状態とドメイン状態の変換を ViewModel に寄せる。

View
ViewModel
Model
binding

動き: View と ViewModel の間で表示状態が同期され、業務状態は Model 側へ渡る。UI 状態の変換点が見える。

利点

  • 複雑な画面状態を View から分離し、テストしやすくする。
  • デスクトップ、モバイル、SPA など状態同期が多い UI に向く。

トレードオフ

状態整理高
バインド複雑性高
追跡容易性中

相性の良いシステムアーキテクチャ

Client-Server3-Tier / N-TierEdge / CDN-Oriented

画面側で状態を持つ構成や、フロント配信をエッジに寄せる構成で、UI 状態と API 状態を分離しやすい。

06

Modular Monolith

単一デプロイのまま、コード内部を明確なモジュール境界で分ける。モノリスとマイクロサービスの中間的な選択肢。

Single Process / Single Deploy
Catalog
Orders
Billing
Shared DB Access
boundary bypass

動き: 通常の呼び出しは公開境界を通る。共有 DB 直アクセスのような近道が増えると、モジュール境界が崩れる。

利点

  • 分散運用の複雑さを避けつつ、コード上の変更影響を局所化できる。
  • 必要になったモジュールを将来サービス化する準備にもなる。

トレードオフ

運用単純性高
規律依存高
部分スケール低

相性の良いシステムアーキテクチャ

Monolithic System3-Tier / N-TierHybrid / Distributed

単一デプロイの利点を残しつつ、移行や将来分割のための境界をコード上に置ける。

07

Domain-Driven Design

業務ドメインを中心に、Bounded Context、ユビキタス言語、集約、ドメインイベントなどでモデルを分ける。

Ordering Context
Billing Context
Inventory Context
ACL
order terms
billing terms
translate model
domain event

動き: 各コンテキストは独自モデルを持ち、イベントや ACL を通じて連携する。言葉とモデルの境界を保つ。

利点

  • 業務の複雑さをモデルとして扱い、チームやサービスの境界を決めやすい。
  • ドメイン知識がコードに現れるため、長期保守で認知負荷を下げやすい。

トレードオフ

業務適合高
分析コスト高
境界誤認リスク高

相性の良いシステムアーキテクチャ

MicroservicesMonolithic SystemEvent-Driven System

Bounded Context をサービスやモジュールの候補にでき、ドメインイベントで境界間の連携を表しやすい。

08

CQRS

Command と Query のモデルを分離する。書き込みの整合性と読み取りの最適化を別々に設計する。

Command
Write Model
Event
Read Model
Query
eventual consistency

動き: 書き込みは Write Model に入り、イベントを通じて Read Model が遅れて更新される。読み取りは最適化済みモデルを直接参照する。

利点

  • 読み取り負荷が大きい場合、クエリ用モデルを用途別に最適化できる。
  • 書き込みのビジネス制約と読み取りの表示要件を分けられる。

トレードオフ

読み取り性能高
結果整合性高
モデル数高

相性の良いシステムアーキテクチャ

Event-Driven SystemMicroservicesEdge / CDN-Oriented

読み取りと書き込みを別スケールにでき、イベント連携やキャッシュ配信と自然に組み合わせられる。

09

Event Sourcing

現在状態だけでなく、状態変化のイベント列を事実として保存する。状態はイベントの再生から復元する。

Command
Event Log
Projection
State View
append only

動き: コマンドからイベントが追記され、Projection がイベントを再生して読み取り用の状態を作る。

利点

  • 監査、履歴、再構築に強く、なぜその状態になったかを追いやすい。
  • 後から新しい Projection を作り、過去イベントから別の見方を生成できる。

トレードオフ

監査性高
設計難度高
イベント進化高

相性の良いシステムアーキテクチャ

Event-Driven SystemData PipelineMicroservices

イベントログを連携や分析に使えるため、イベント中心のシステムや履歴分析と相性が良い。

10

Pipeline Architecture

入力を複数の段階に通し、各段階が変換、検証、集約、出力など一つの責務を担う。

Input
Parse
Validate
Transform
Output
stage isolation

動き: データが段階を順に通過する。重い段階だけを複数化すれば、流量の詰まりを緩和できる。

利点

  • 各段階の責務を小さくし、再利用、テスト、差し替えをしやすい。
  • データ変換、画像処理、ETL、CI/CD など直列処理に向く。

トレードオフ

段階分離高
全体遅延中
失敗再処理高

相性の良いシステムアーキテクチャ

Data PipelineServerlessEvent-Driven System

段階ごとにキュー、関数、ジョブへ分けやすく、処理量に応じたスケールにも合わせやすい。

11

Plugin Architecture

安定したコアに拡張ポイントを用意し、追加機能を plugin として差し込む。

Core
Plugin A
Plugin B
Plugin C
Plugin D
extension points

動き: plugin はコアの拡張ポイントを通じて接続される。コア契約が安定していれば追加機能を差し替えやすい。

利点

  • 個別機能や顧客別拡張をコアから分離できる。
  • 拡張開発者とコア開発者の作業境界を作りやすい。

トレードオフ

拡張性高
API 固定負荷高
互換性管理高

相性の良いシステムアーキテクチャ

SOAHybrid / DistributedEdge / CDN-Oriented

環境差分や外部連携差分を拡張として扱えるため、複数環境や顧客別構成に対応しやすい。

12

Microkernel Architecture

最小限の中核機能を kernel に集約し、周辺機能を extension として追加する。Plugin Architecture の一種として扱われることもある。

Extensions
Kernel
Ext A
Ext B
Ext C
Ext D
kernel contract

動き: 小さな kernel が共通機構を提供し、extension が機能を追加する。kernel が太ると拡張性が落ちる。

利点

  • 中核を安定させ、周辺機能を独立して追加、削除、更新しやすい。
  • IDE、OS 周辺、業務プラットフォームなど拡張前提の製品に向く。

トレードオフ

中核安定性高
契約設計高
性能間接性中

相性の良いシステムアーキテクチャ

Edge / CDN-OrientedHybrid / DistributedSOA

共通中核と環境別拡張の形に分けると、地域差、顧客差、連携先差分を extension として扱いやすい。
