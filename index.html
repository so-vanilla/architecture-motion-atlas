Boundary

混同しないための境界

同じ「アーキテクチャ」でも、何を設計対象にしているかが違う。 システム側の選択は実行時の構成に効き、ソフトウェア側の選択は変更時の構造に効く。

システムアーキテクチャ

ネットワーク上の配置、デプロイ単位、通信方式、可用性、スケール方式、 障害分離、運用境界を決める。図ではリクエスト、イベント、障害、レプリカ増減を動かす。

  • 主な問い: どこで動き、どう接続し、どう増減するか
  • 代表例: Microservices、Serverless、Edge、Event-Driven System
  • 失敗しやすい点: 分散化による運用複雑性を過小評価すること

ソフトウェアアーキテクチャ

コード内部の依存方向、責務分離、ドメイン境界、状態更新、テスト容易性、 変更影響範囲を決める。図では依存方向、更新伝播、境界越え、投影の遅延を動かす。

  • 主な問い: どの責務をどこに置き、何に依存させるか
  • 代表例: Clean、Hexagonal、CQRS、DDD、Modular Monolith
  • 失敗しやすい点: 形式を導入しても境界の意味を保てないこと

Pairing Map

相性の方向性

相性は絶対ではない。ここでは「その組み合わせにすると、制約と利点が自然に噛み合いやすい」 という意味で扱う。

Microservices → Hexagonal / DDD / CQRS サービス境界ごとにドメインと外部 adapter を分け、読み書きの負荷差にも対応しやすい。
3-Tier → Layered / MVC 画面、アプリケーション処理、データ層の分離が、配置上の階層と対応する。
Event-Driven System → Event Sourcing / CQRS イベントを主語にした連携と、イベントから読み取りモデルを作る設計が揃う。
Serverless → Pipeline / Hexagonal イベント入力と外部サービス依存を明確に分けると、関数の粒度を保ちやすい。
Monolithic System → Modular Monolith / Layered 単一デプロイの単純さを保ちながら、コード上の境界で変更影響を抑えられる。
Hybrid / Edge → Plugin / Microkernel 中核を固定し、環境差分や地域別拡張だけを差し替える用途に向く。

Trade-off Axes

比較軸

図の中では、利点だけでなく負荷が増える軸もバーで示す。スケールや疎結合が強くなるほど、 運用、観測、整合性、設計規律の負担が増える場合がある。
