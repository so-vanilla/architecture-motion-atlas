System Architecture

システムアーキテクチャ

ここで扱うのは、実行時の配置、通信、デプロイ単位、スケール、可用性、運用境界である。 コード内部の依存設計ではなく、システムが外からどう見え、どう増減し、どう壊れるかを中心に見る。

  • Hybrid / Distributed System

01

Client-Server

クライアントがリクエストを送り、サーバーが処理と状態管理を担う基本形。通信の向きが明確で、構成を把握しやすい。

Client
Server
State
server capacity

動き: リクエストが一点のサーバーへ集まり、負荷が上がるとサーバー側が熱を持つ。応答は同期的に戻る。

利点

  • 責務の位置が単純で、認証、状態管理、データ保護を中央に寄せやすい。
  • 小規模から中規模の業務システムで構成理解と障害調査がしやすい。

トレードオフ

単純性高
集中リスク高
独立スケール低

相性の良いソフトウェアアーキテクチャ

MVCLayered

要求、制御、永続化の流れが同期的なので、画面入力から処理、保存までを階層化しやすい。

02

3-Tier / N-Tier

Presentation、Application、Data の階層を別配置にする。各階層の責務とスケール単位を分けられる。

Presentation
Application
Data
Application
app tier scales

動き: 要求は階層を順に進む。アプリケーション層だけを増やし、画面層やデータ層とは別に負荷へ対応する。

利点

  • 層ごとの責務、ネットワーク境界、セキュリティ境界を合わせやすい。
  • Web/AP/DB の典型構成として、運用設計や監視設計の定石が多い。

トレードオフ

階層分離高
往復遅延中
DB集中高

相性の良いソフトウェアアーキテクチャ

LayeredMVCMVVM

配置上の層とコード上の層が対応するため、責務分離を説明しやすく、チーム分担もしやすい。

03

Monolithic System

主要機能を単一アプリケーションとしてデプロイする。通信はプロセス内呼び出しが中心で、リリース単位も一つになる。

Single Deployable
Orders
Billing
Users
Inventory
DB
one release can touch all

動き: 外部通信は単一アプリへ入り、内部では高速に呼び出される。変更や障害は同じデプロイ単位へ広がりやすい。

利点

  • 分散システムの複雑さが少なく、ローカルトランザクションを使いやすい。
  • 初期開発、単一チーム、強い一貫性が必要な業務に向く。

トレードオフ

開発速度初期高
変更波及高
部分スケール低

相性の良いソフトウェアアーキテクチャ

Modular MonolithLayeredDDD

デプロイ単位は一つでも、コード境界を保てば変更影響を抑えられる。ドメイン境界がない単なる巨大アプリとは別物として扱う必要がある。

04

Microservices

業務能力ごとに独立したサービスとしてデプロイする。サービスごとにスケール、リリース、障害分離を調整できる。

Gateway
Auth
Orders
Orders
Payment
Auth DB
Order DB
Pay DB
scale hot service
network + tracing cost

動き: 注文サービスだけレプリカが増え、決済の障害は局所化される。ただし通信経路と観測対象は増える。

利点

  • サービス単位で独立デプロイ、独立スケール、技術選択ができる。
  • チーム境界とサービス境界を合わせると、変更の待ち合わせを減らせる。

トレードオフ

独立性高
運用複雑性高
整合性難度高

相性の良いソフトウェアアーキテクチャ

HexagonalCleanDDDCQRS

サービス境界ごとにドメインを閉じ、外部通信や永続化を adapter として扱うと、分散の複雑さをコード内部へ漏らしにくい。

05

SOA

企業内の複数システムを再利用可能なサービスとして統合する。ESB や共通契約を使い、部門横断の連携を整理することが多い。

Service Bus
CRM
ERP
Billing
Legacy
central contract

動き: 各システムは共通バスへ接続し、変換やルーティングを中央に寄せる。中央の契約が強くなるほど変更調整が重くなる。

利点

  • 既存システムや部門システムを共通契約で接続し、再利用しやすくする。
  • 変換、監査、ポリシー適用を統合ポイントで扱いやすい。

トレードオフ

統合性高
中央依存高
変更俊敏性中

相性の良いソフトウェアアーキテクチャ

LayeredHexagonalPlugin

外部契約や変換ロジックを境界に閉じ込めると、中心の業務ロジックを連携方式から守りやすい。

06

Event-Driven System

状態変化をイベントとして発行し、複数の購読者が非同期に処理する。送信側と受信側を時間的に分離する。

Producer
Broker
Consumer A
Consumer B
Consumer C
backlog / retry

動き: イベントがブローカーに積まれ、購読者は自分の速度で処理する。遅い購読者にはバックログができる。

利点

  • 送信側と受信側を疎結合にし、購読者追加で機能拡張しやすい。
  • ピークをキューで吸収し、非同期処理によりユーザー待ち時間を減らせる。

トレードオフ

疎結合高
追跡難度高
順序/重複高

相性の良いソフトウェアアーキテクチャ

Event SourcingCQRSPipeline

イベントを記録、投影、変換するコード構造と噛み合う。重複処理や結果整合性を設計として明示しやすい。

07

Serverless / FaaS

イベントに応じて関数やマネージドサービスが起動する。サーバー管理をクラウド側に寄せ、使用量に応じて増減する。

HTTP
Queue
Function
Function
Function
Managed DB
cold start

動き: HTTP やキューイベントが関数を起動し、負荷に応じて関数インスタンスが増える。初回起動には待ち時間が出ることがある。

利点

  • インフラ運用を小さくし、イベント量に応じた自動スケールを得やすい。
  • 小さな自動処理、画像変換、バッチ、API の一部に向く。

トレードオフ

自動スケール高
ベンダ依存高
実行制約高

相性の良いソフトウェアアーキテクチャ

PipelineHexagonalCQRS

関数は入力と出力を小さく保つ必要があるため、変換段階、外部 adapter、読み書きモデルの分離を明確にする設計と相性が良い。

08

Edge / CDN-Oriented

ユーザーに近いエッジで静的配信、キャッシュ、軽量処理を行う。オリジンへの到達回数と距離を減らす。

User JP
User EU
Edge JP
Edge EU
Origin
cache hit
cache miss

動き: 多くの要求は近いエッジで止まり、ミスした要求だけがオリジンへ進む。レイテンシとオリジン負荷を下げる。

利点

  • 地理的に分散したユーザーへ低遅延で配信しやすい。
  • 静的コンテンツや読み取り中心 API の負荷をオリジンから逃がせる。

トレードオフ

低遅延高
無効化難度高
状態管理高

相性の良いソフトウェアアーキテクチャ

CQRSMVVMMicrokernel

読み取りモデルを分けるとキャッシュしやすい。画面状態やエッジ固有処理を薄い境界として扱うと、中核ロジックを汚しにくい。

09

Data Pipeline / Analytics Platform

業務システム、ログ、イベントを集め、変換し、分析用ストアへ届ける。処理はバッチまたはストリームになる。

動き: 複数ソースからデータが取り込まれ、変換を通って分析用ストアへ入る。バッチ処理では可視化まで遅延がある。

利点

  • 業務負荷から分析負荷を分離し、横断的な意思決定に使えるデータを作れる。
  • 変換段階を分けることで、品質チェックや再処理がしやすい。

10

Hybrid / Distributed System

オンプレミス、クラウド、複数地域など、異なる環境を接続して動かす。段階移行や規制要件に対応しやすい。

動き: 環境間の通信は専用線や VPN を通る。リンク障害や遅延を前提に、キューや再試行で吸収する。

利点

  • 既存資産や規制対象データを残しつつ、クラウド側で新機能やスケールを得られる。
  • 移行期間や複数地域対応で、段階的な切り替えがしやすい。
